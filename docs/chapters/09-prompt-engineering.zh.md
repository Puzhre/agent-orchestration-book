# 第九章 Prompt工程：铁律与约束

> 硬编排确保Agent持续运行，但运行着做什么？软编排的第一道防线是Prompt——通过精心设计的指令约束Agent行为。

## 9.1 为什么Prompt需要"铁律"

AI Agent有一个致命特性：**它会修改自己的约束**。

当你给Agent写了一条规则"不要删除文件"，Agent在遇到阻碍时可能：

1. 直接删掉这条规则
2. 重新解释规则使其不再适用
3. 找到绕过规则的路径

这不是理论推演，而是实战中反复出现的问题。agent_prompt.txt顶部和底部各有一个核心规则块，就是因为Agent曾经删掉了规则——加了双重块后，即使删了一个，另一个还在。

## 9.2 铁律设计模式

### 模式一：双重核心规则块（Double Iron Law Block）

```
════════════════════════════════════════
CORE RULE BLOCK — 铁律（不可删除）
════════════════════════════════════════
1. 绝不删除项目文件
2. 绝不修改模型配置
════════════════════════════════════════
... (其他 prompt 内容) ...
════════════════════════════════════════
CORE RULE BLOCK (END) — 核心规则块结束
════════════════════════════════════════
```

**原理**：顶部声明+底部确认，即使Agent修改了中间内容，两端的铁律仍在
**局限性**：无法阻止Agent完全重写prompt文件

### 模式二：外部守护（External Guardian）

```
# 每5分钟检查核心规则块完整性
rule_guard() {
    local prompt_file="agent_prompt.txt"
    if ! grep -q "CORE RULE BLOCK" "$prompt_file"; then
        git checkout "$prompt_file"  # 从git恢复
        warn_agent "铁律被删除，已自动恢复"
    fi
}
```

**原理**：Agent无法修改编排器守护进程的代码，守护进程定期检查并恢复prompt
**优势**：Agent与守护者不在同一进程空间，物理隔离

### 模式三：分层约束（Layered Constraints）

| 层级 | 约束方式 | 可被Agent修改？ | 示例 |
|------|---------|---------------|------|
| L0 | 系统prompt | 否（由编排器注入） | "你是XX项目的Agent" |
| L1 | 核心规则块 | 是（在用户消息中） | "绝不删除文件" |
| L2 | MISSION注入 | 是 | "你的使命是..." |
| L3 | SPRINT驱动 | 是 | "当前sprint目标..." |
| L4 | 创意提示 | 是 | "你可以尝试..." |

**关键洞察**：越重要的约束应该放在越低的层级（越难被修改）。

## 9.3 MISSION注入：持续的方向感

Agent在长对话中会逐渐遗忘初始指令。MISSION机制在每次对话开始时注入一段简洁的使命描述：

```bash
PROJECT_MISSION="你是一个严格的产品创意守门人。少而精，宁杀勿纵。
第一步永远先检查active/是否满3个。毕业必须92分+50轮+各维度7.5+。"
```

**对比**：

| 项目 | MISSION载体 | 注入频率 | 效果 |
|------|-----------|---------|------|
| Tmux-Orchestrator | CLAUDE.md | Agent自己维护 | 弱 |
| Overstory | 初始prompt | 不再注入 | 弱 |
| agency-agents-zh | Markdown prompt文件 | 每次对话 | 强 |

## 9.4 SPRINT驱动：结构化的工作目标

没有明确目标的Agent会反复做同一件事。SPRINT.md给出结构化的当前目标：

```markdown
# Sprint状态
## 当前目标
- 分析 CrewAI 的角色编排机制
## 进行中
- [ ] 克隆仓库
- [ ] 提取架构模式
## 已完成
- [x] 5个项目分析
```

**与MISSION的区别**：MISSION是长期方向，SPRINT是短期目标。两者配合使用。

## 9.5 反模式：Prompt的常见陷阱

### 陷阱一：规则爆炸

```
❌ 50条规则的prompt
→ Agent只能记住前10条和最后5条
→ 中间的35条形同虚设

✅ 5条铁律 + MISSION + SPRINT
→ 铁律不可违反
→ MISSION给方向
→ SPRINT给目标
```

### 陷阱二：正面规则vs负面规则

```
❌ "写好代码" — 太模糊，Agent各有理解
✅ "每次改完必须跑测试" — 具体可验证

❌ "不要出错" — 无法执行
✅ "连续3次失败就升级" — 可执行策略
```

### 陷阱三：一次性Prompt

```
❌ 只在启动时给一次长prompt
→ 50轮对话后Agent已经遗忘80%

✅ 每轮注入MISSION + 定期检查铁律
→ 上下文窗口中的约束始终存在
```

## 9.6 小结

Prompt工程是软编排的基石。好的Prompt不是一次性的长文，而是：

1. **铁律**：不可违反的底线约束（双重块+外部守护）
2. **MISSION**：持续的方向感（每次注入）
3. **SPRINT**：结构化的短期目标（随进度更新）
4. **分层**：重要约束放低层级，防止被覆盖
